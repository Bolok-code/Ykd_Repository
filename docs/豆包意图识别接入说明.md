# Spring AI + 豆包意图识别接入说明

## 1. 目标

微信机器人不再通过“包含天气、画图、语音”等关键词或正则表达式判断用户意图，
而是先调用一个独立的小模型，把自然语言转换成项目可校验的结构化结果。

需要区分两个概念：

- 入站消息格式：图片、语音、文字，由 iLink SDK 的 `ImageItem`、`VoiceItem`
  和 `TextItem` 字段确定，不让大模型猜。
- 用户语义：普通聊天、天气查询、文生图，以及希望文字还是语音回复，由豆包判断。

## 2. 完整流程

```text
微信消息
  ├─ 图片 ───────────────> 百炼视觉理解
  ├─ 语音 ─> VoiceItem.text ─┐
  └─ 文字 ───────────────────┤
                             v
                    IntentClassifier
                             |
                 SpringAiDoubaoIntentClassifier
                             |
                火山方舟 Chat Completions
                             |
                     IntentDecision
           ┌─────────────────┼──────────────────┐
           v                 v                  v
       TEXT_CHAT       WEATHER_QUERY     IMAGE_GENERATION
           |                 |                  |
       DeepSeek         WeatherService      百炼文生图
           └─────────────────┴──────────────────┘
                             |
                  ReplyMode=TEXT 或 VOICE
                             |
                 iLink 发文字 / 百炼 TTS 发文件
```

“用语音回答明天杭州天气”不会被简单归类成一个 `VOICE_REPLY`。
模型会返回：

```json
{
  "intent": "WEATHER_QUERY",
  "replyMode": "VOICE",
  "confidence": 0.98,
  "city": "杭州",
  "targetDate": "2026-07-21",
  "requestText": "明天杭州天气"
}
```

这样天气是业务能力，语音只是输出形式，两者不会互相覆盖，也不需要二次识别。

## 3. 主要类

```text
com.clitoolbox/
├─ intent/
│  ├─ IntentType.java
│  ├─ ReplyMode.java
│  ├─ IntentDecision.java
│  └─ IntentClassifier.java
├─ ai/intent/
│  ├─ IntentAiConfiguration.java
│  └─ SpringAiDoubaoIntentClassifier.java
├─ config/
│  ├─ IntentAiProperties.java
│  └─ IntentAiConfig.java
└─ ilink/
   ├─ router/ILinkMessageRouter.java
   └─ service/impl/ILinkServiceImpl.java
```

- `IntentClassifier`：业务层依赖的稳定接口。以后换模型，只需增加新的实现。
- `SpringAiDoubaoIntentClassifier`：使用 Spring AI 的 `OpenAiChatModel`
  调用火山方舟，编写分类提示词并解析 JSON。
- `IntentDecision`：模型结果的项目内部表示。包含业务意图、回复形式、置信度、
  城市、日期和实际问题。
- `IntentAiConfig`：检查 API Key、地址、模型、超时和置信度阈值，日志中不输出密钥。
- `IntentAiConfiguration`：创建独立的 `doubaoIntentChatModel` Bean。
- `ILinkMessageRouter`：图片直接分流；文字调用 `IntentClassifier` 后分配给具体业务。
- `ILinkServiceImpl`：执行天气、DeepSeek、文生图或 TTS，并通过 iLink 发回微信。

项目同时存在 DeepSeek 和豆包两个 `ChatModel`，因此分别使用
`deepSeekChatModel`、`doubaoIntentChatModel` Bean 名和 `@Qualifier`，
避免 Spring 不知道该注入哪个模型。

## 4. 配置

公开配置位于 `src/main/resources/application.yml`：

```yaml
app:
  intent-ai:
    api-key: ${ARK_API_KEY:}
    base-url: ${ARK_BASE_URL:https://ark.cn-beijing.volces.com/api/v3}
    model: ${ARK_INTENT_MODEL:doubao-seed-2-0-mini-260428}
    max-tokens: ${ARK_INTENT_MAX_TOKENS:350}
    timeout-seconds: ${ARK_INTENT_TIMEOUT_SECONDS:10}
    confidence-threshold: ${INTENT_CONFIDENCE_THRESHOLD:0.70}
```

真实密钥只能使用环境变量，或填写在已经被 `.gitignore` 排除的
`config/application-local.yml`：

```yaml
app:
  intent-ai:
    api-key: 你的火山方舟APIKey
```

不要把真实密钥写入 `application.yml`、Java 源码、测试或 Git。

## 5. 当前还差的控制台动作

真实请求已经完成鉴权，但火山方舟返回：

```text
ModelNotOpen
```

因此还需要在火山方舟控制台开通以下在线推理模型：

```text
Doubao-Seed-2.0-mini
版本：260428
模型 ID：doubao-seed-2-0-mini-260428
```

这是模型服务开通动作，可能启用按 Token 后付费，应由账号所有者在确认计费方式后完成。
API Key 本身不能代替模型开通。

开通后，在项目根目录执行真实测试：

```powershell
mvn "-Dtest=SpringAiDoubaoIntentClassifierLiveIT" test
```

该测试验证五类真实表达：

```text
今天天气好好啊                       -> TEXT_CHAT
今天去杭州玩了，天气真好             -> TEXT_CHAT
明天杭州天气怎么样                   -> WEATHER_QUERY
帮我画一只戴宇航员头盔的橘猫         -> IMAGE_GENERATION
请用语音回答：李白是谁               -> TEXT_CHAT + VOICE
```

普通 `mvn test` 不会运行这个真实 API 测试，也不会消耗方舟额度。

## 6. 安全与降级

- 置信度低于 `0.70` 时回落普通文字聊天。
- 豆包超时、限流、返回无效 JSON 或服务异常时回落普通文字聊天。
- 天气意图缺少城市时提示用户补充城市，不调用天气 API。
- 模型输出的枚举、日期、置信度和必要字段必须经过 Java 校验。
- 图片消息不调用意图模型，直接进入视觉理解。
- `/clear`、`/new` 等精确项目命令不调用意图模型。
- 每条自然语言消息最多只做一次意图判断，避免重复回复和重复计费。

## 7. 测试

本地测试覆盖：

- Spring AI 请求路径、Bearer 鉴权头、模型、超时参数和 `thinking=disabled`。
- JSON 结果、Markdown 代码块结果和非法结果的解析。
- 低置信度安全回落。
- 豆包不可用时 Router 回落普通聊天。
- 图片和斜杠命令绕过意图模型。
- `WEATHER_QUERY + VOICE` 两个维度同时存在。
- Spring Boot 中 DeepSeek、豆包两个 `ChatModel` 正确装配。
